## 1. 프로젝트 개요

Antigravity IDE를 사용하면서 한 가지 불편함이 있었다. 다른 IDE들과 달리 Quota 사용량을 직관적으로 확인할 수 없다는 점이었다.

2026년 2월 19일 업데이트로 Antigravity에 Settings → Models 화면이 추가되면서 모델별 Quota 사용량을 확인할 수 있게 됐다. 하지만 확인할 때마다 설정 화면으로 직접 이동해야 한다는 불편함은 여전히 남아있었다.

물론 이미 이를 해결하기 위한 Extension들은 존재했다. 하지만 바로 설치해서 사용하기보다, 먼저 GitHub에 공개된 코드와 AI를 활용해 동작 방식을 분석해봤다. 어떤 방식으로 Quota 데이터를 가져오는지 궁금하기도 했고, 구조적으로 문제가 있을 수 있겠다는 생각이 들었기 때문이다.

분석 결과, **내 기준에서** 문제가 예상되는 부분을 발견했다. 물론 문제가 되지 않는다고 생각할 수도 있지만, 나는 정책 위반과 인증 정보 접근이 문제가 될 것이라고 판단했다. 그렇다면 이 문제를 피하면서도 같은 기능을 구현할 수 있는 방법이 있지 않을까 생각했고, 그게 직접 만들어보게 된 계기가 됐다.

---

## 2. 기존 Extension의 방식과 문제점

다운로드 수가 많았던 Extension들의 동작 방식을 비교해보니, 공통적으로 다음과 같은 흐름으로 동작하고 있었다.

1. Antigravity의 language server 프로세스를 스캔
2. 실행 인자에서 `--csrf_token` 추출
3. 해당 토큰으로 내부 API 호출

동작 자체는 문제없다. 하지만 이 방식에는 두 가지 근본적인 문제가 있었다.

### 1) 서비스 이용약관(ToS) 위반

이 Extension들은 Antigravity가 공식적으로 공개하지 않은 내부 API를 역공학해서 사용한다. 이는 대부분의 SaaS 서비스 이용약관에서 금지하는 방식이다.

정책을 위반하면서까지 Quota 사용량을 확인하고 싶지는 않았다.

### 2) 인증 정보 접근 문제

CSRF 토큰은 단순한 값이 아니라 사용자의 인증 상태를 보호하는 보안 요소다. 이를 프로세스 인자에서 추출해 재사용하는 구조는, 사용자 인증 정보를 우회적으로 활용하는 방식과 본질적으로 다르지 않다.

특히 Extension은 외부 코드이기 때문에, 악의적인 업데이트나 공급망 공격(Supply Chain Attack)이 발생하면 토큰 탈취로 이어질 수 있는 구조다.

## 3. 접근 방식의 전환

Settings → Models 화면에 Quota 데이터가 표시된다면, 그 데이터는 어딘가에서 오고 있을 것이다.

> "이 데이터는 어디서 오는 걸까?"

---

## 4. 핵심 아이디어

Antigravity는 VS Code 기반의 포크다. 그리고 VS Code는 다양한 상태 정보를 로컬 SQLite DB인 `state.vscdb`에 저장한다.

확인해보니, Quota 사용량 데이터 역시 이 DB에 저장되고 있었다. 단, 단순한 텍스트나 JSON이 아니라 **Base64로 인코딩된 Protobuf 바이너리** 형태로 저장되어 있었다.

즉, 굳이 프로세스를 스캔하거나 내부 API를 호출하지 않아도, **이미 로컬에 저장된 데이터를 읽기만 하면 된다.**

---

## 5. 구현 방식

이 프로젝트의 핵심 방향은 **정책 위반과 보안 위험성을 최대한 줄이는 것**이었다. 구현은 크게 세 단계로 나뉜다.

### 1단계 — 안전한 파일 접근

`state.vscdb`는 IDE가 실시간으로 쓰고 있는 파일이다. 직접 열면 충돌이 발생할 수 있기 때문에, 임시 복사본(`ag_quota_read.vscdb`)을 만들고 복사본에서만 읽는다.

읽기가 끝나면 복사본은 즉시 삭제한다. 에러가 발생하더라도 반드시 삭제되도록 `finally` 블록으로 보장했다. 임시 파일이 남아 디스크를 오염시키는 상황을 방지하기 위한 판단이었다.

### 2단계 — 디코딩

DB에서 꺼낸 값은 Base64로 인코딩된 Protobuf 바이너리다. 외부 라이브러리 의존성을 최소화하는 방향으로 직접 Protobuf 파서를 구현했다. 다만 Protobuf 스펙 전체를 구현한 것이 아니라, Quota 데이터를 읽는 데 필요한 필드만 파싱하는 수준으로 제한해 복잡도를 통제했다.

### 3단계 — UI 표시

파싱한 결과를 두 가지 방식으로 표시한다.

- **Status Bar**: 현재 모델의 Quota 수치와 아이콘을 표시. 사용량에 따라 아이콘과 색상이 변경된다.
- **Webview Panel**: 모델별 원형 progress ring 카드를 그리드 레이아웃으로 배치해 전체 현황을 한눈에 파악할 수 있다.

또한 30초마다 백그라운드 폴링을 통해 자동으로 갱신되며, Antigravity 창이 포커스를 되찾을 때도 자동으로 갱신된다.

## 7. 기존 방식과의 차이

기존 Extension들이 프로세스를 스캔하고 내부 API를 호출하는 구조였다면, 이 프로젝트는 IDE가 로컬에 기록한 데이터를 직접 읽는 구조다. Quota 사용량을 보여준다는 결과는 같지만, 데이터를 가져오는 방식 자체가 다르다.

정책 위반과 보안 리스크가 될 수 있는 부분을 피하는 것이 이 프로젝트의 출발점이었고, 그 결과로 접근 방식이 달라졌다.

- ❌ 내부 API 호출 없음
- ❌ CSRF 토큰 추출 없음

## 8. 한계점

`state.vscdb` 기반 접근 방식은 정책과 보안 측면에서 안전하지만, 명확한 한계가 있다.

기존 Extension들은 내부 API를 직접 호출하기 때문에 실시간으로 정확한 Quota 수치를 가져올 수 있었다. 반면 이 프로젝트는 `state.vscdb`에 저장된 값을 읽는 방식이기 때문에, **DB가 업데이트되지 않은 경우 실제 사용량과 표시값이 다를 수 있다.** 심한 경우 Quota를 모두 소진했더라도 DB에 반영되기 전이라면 이전 값을 그대로 보여줄 수밖에 없다.

또한 Antigravity가 내부 데이터 구조나 포맷을 변경하면 Extension이 정상적으로 동작하지 않을 수 있다는 점도 한계다.

다만, Antigravity가 Settings → Models 화면에서 Quota 사용량을 공식적으로 제공하기 시작했다는 점은 긍정적인 신호다. 조만간 다른 IDE처럼 Quota를 직관적으로 확인할 수 있는 기능이 공식 지원될 가능성이 높다.

그 전까지, 정책과 보안을 지키면서 한눈에 Quota 현황을 파악할 수 있는 실용적인 대안을 만들고자 했다.

---

## 9. 마무리

같은 목적이라도 무엇을 먼저 두느냐에 따라 구현 방식이 달라지는 것을 체감했다. 여러 서비스들을 보면 같은 주제여도 어떤 철학을 중심에 두느냐에 따라 기능과 방향이 달라지는데, Extension도 결국 다르지 않은 것 같다.

기존 Extension들과 이 프로젝트가 추구하는 바는 같다. Quota 사용량을 편하게 확인하는 것.

다만 우선순위가 달랐다. 기존 Extension들은 정확한 실시간 데이터를 최우선으로 두었고, 어떻게 보면 그게 사용자 입장에서 원래의 의도에 더 충실한 방식일 수도 있다. 반면 나는 정책과 보안 리스크를 피하는 것을 먼저 두었고, 그 결과로 정확도에서 일부 타협이 생겼다.

정책과 보안 측면에서 걱정 없이 쓸 수 있는 Extension을 만들고 싶었고, 내가 추구하는 방향대로 잘 만든 것 같다.

Antigravity에서 공식적으로 지원하기 전까진 계속 유지보수와 업데이트를 진행할 예정이다. v0.2.0에서는 설정 기능을 추가할 계획이다. 언어 설정(한국어/영어), 표시 모델 수(기본/상세), 갱신 주기 설정 등을 지원할 것 같다.
